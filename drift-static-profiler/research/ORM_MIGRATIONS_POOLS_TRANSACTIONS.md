# ORM Schema Migrations, Connection Pools, and Transaction Management — Detector Catalog

Research output for drift-static-profiler. Three categories of database concerns:
(A) schema migrations, (B) connection pool configuration, (C) transaction
management. Every entry includes tree-sitter-friendly shape, severity/effort,
and license guidance for any source to port from.

---

## PART A — Schema Migration Tools & Anti-Patterns

### A.0 Gold-standard rule catalogs to port

**squawk** (`sbdchd/squawk`, GPL-3.0) — most complete migration linter for Postgres. Rules to port (full catalog at https://squawkhq.com/docs/rules):

| Rule | Bad pattern | Severity |
|------|-------------|----------|
| `adding-required-field` | `ALTER TABLE ADD COLUMN ... NOT NULL` without default | High |
| `adding-field-with-default` | `ADD COLUMN ... DEFAULT ...` on Postgres <11 (table rewrite) | High |
| `adding-foreign-key-constraint` | `ADD CONSTRAINT ... FOREIGN KEY` without `NOT VALID` | High |
| `adding-not-nullable-field` | `ALTER COLUMN ... SET NOT NULL` (full table scan, ACCESS EXCLUSIVE) | High |
| `adding-serial-primary-key-field` | `ADD COLUMN id serial PRIMARY KEY` (rewrite + lock) | High |
| `ban-char-field` | `char(n)` columns | Low |
| `ban-drop-column` | `DROP COLUMN` (breaks rolling deploys) | High |
| `ban-drop-database` / `ban-drop-table` | obvious | High |
| `ban-drop-not-null` | dropping NOT NULL breaks clients expecting non-null | Medium |
| `changing-column-type` | `ALTER COLUMN ... TYPE` (rewrite + lock) | High |
| `constraint-missing-not-valid` | `ADD CONSTRAINT` without `NOT VALID` then `VALIDATE` | High |
| `disallowed-unique-constraint` | `ADD CONSTRAINT ... UNIQUE` (uses `ACCESS EXCLUSIVE`) — use `CREATE UNIQUE INDEX CONCURRENTLY` + `ADD CONSTRAINT USING INDEX` | High |
| `prefer-bigint-over-int` / `prefer-bigint-over-smallint` | `int`/`smallint` keys exhaust | Medium |
| `prefer-identity` | `serial` instead of `GENERATED ... AS IDENTITY` | Low |
| `prefer-robust-stmts` | DDL not wrapped in transaction or missing `IF NOT EXISTS` | Medium |
| `prefer-text-field` | `varchar(n)` | Low |
| `prefer-timestamptz` | `timestamp` without timezone | Medium |
| `renaming-column` / `renaming-table` | breaks deployed clients | High |
| `require-concurrent-index-creation` | `CREATE INDEX` without `CONCURRENTLY` | High |
| `require-concurrent-index-deletion` | `DROP INDEX` without `CONCURRENTLY` | Medium |
| `transaction-nesting` | `BEGIN` inside migration that's already wrapped | Medium |
| `ban-concurrent-index-creation-in-transaction` | `CREATE INDEX CONCURRENTLY` inside `BEGIN` (fails at runtime) | High |

**License note:** squawk is GPL-3.0 — do NOT copy code. Rule lists are facts, not copyrightable; re-implement detection in Rust against tree-sitter or `libpg_query` (MIT, already a drift dep via `pg_query` crate).

**strong_migrations** (`ankane/strong_migrations`, MIT — Ruby/ActiveRecord) — port verbatim. Rules: `removing_a_column`, `adding_a_column_with_a_default`, `backfilling_data`, `adding_a_foreign_key`, `adding_a_check_constraint`, `adding_a_json_column`, `setting_not_null_on_existing_column`, `executing_sql`, `adding_an_index_non_concurrently`, `adding_a_reference`, `mismatched_foreign_key_type`, `adding_a_stored_generated_column`, `renaming_a_column`, `renaming_a_table`, `changing_the_type_of_a_column`.

**pgroll** (Apache-2.0) and **gh-ost** / **pt-online-schema-change** define the "safe migration" primitives; their operation lists tell us when raw DDL is dangerous.

### A.1 Alembic (SQLAlchemy)

File heuristic: `*/alembic/versions/*.py` or `from alembic import op`.

| Rule | Bad pattern |
|------|-------------|
| `alembic-add-column-not-null-no-default` | `op.add_column('t', sa.Column('c', sa.Integer(), nullable=False))` without `server_default` |
| `alembic-rename-table-no-compat` | `op.rename_table('old','new')` not preceded by view/synonym |
| `alembic-drop-column` | `op.drop_column` without `ignored_columns` in models |
| `alembic-raw-execute-ddl` | `op.execute("ALTER TABLE ...")` bypasses lint |
| `alembic-multi-head` | repository has >1 file with the same `down_revision` |
| `alembic-autogenerate-unreviewed` | `# ### commands auto generated by Alembic - please adjust! ###` left intact |
| `alembic-runpython-no-reverse` | `op.execute` data backfill with `def downgrade(): pass` |

### A.2 Django migrations

File heuristic: `*/migrations/*.py` containing `class Migration(migrations.Migration)`.

| Rule | Bad pattern |
|------|-------------|
| `django-add-field-not-null-no-default` | `migrations.AddField(..., field=models.IntegerField(null=False))` with no `default=` |
| `django-runpython-no-reverse` | `migrations.RunPython(forward)` (single arg) |
| `django-mixed-schema-and-runpython` | `RunPython` in same `operations = [...]` list as schema op |
| `django-atomic-false-missing-on-concurrent-index` | `AddIndex` for Postgres without `atomic = False` class attribute |
| `django-alter-field-changes-type` | `AlterField` where field type differs from prior migration (multi-file diff; Large effort) |

### A.3 Flyway / Liquibase

File heuristic: `V\d+__.*\.sql`, `R__.*\.sql`, `db/changelog/**/*.{xml,yaml,yml,json,sql}`.

| Rule | Bad pattern |
|------|-------------|
| `flyway-modified-versioned-migration` | Any change to a `V*__*.sql` already in `schema_history` table |
| `flyway-repeatable-checksum-bait` | `R__*.sql` with only timestamps/comments — repeatable will never re-fire |
| `flyway-baseline-on-migrate-true` | `flyway.baselineOnMigrate=true` in non-bootstrap config |
| `liquibase-changeset-no-author` | `<changeSet id="..." />` without `author=` |
| `liquibase-multiple-changes-in-changeset` | `<changeSet>` with multiple `<addColumn>` / `<sql>` |
| `liquibase-runOnChange-on-DDL` | `runOnChange="true"` on non-CREATE-OR-REPLACE |

### A.4 Prisma / TypeORM / GORM — the "synchronize" footgun cluster

| Rule | Bad pattern |
|------|-------------|
| `typeorm-synchronize-true` | `synchronize: true` in `DataSource` outside env guard |
| `typeorm-migrationsRun-with-synchronize` | both flags true |
| `gorm-automigrate-prod` | `db.AutoMigrate(&User{}, ...)` in `main` / `init` without env guard |
| `prisma-db-push-in-script` | `prisma db push` in `package.json` `scripts.deploy`/`scripts.start` |
| `prisma-migrate-dev-in-ci` | `prisma migrate dev` in CI YAML (must be `migrate deploy`) |
| `prisma-shadow-db-missing` | `shadowDatabaseUrl` not set with cloud Postgres |

### A.5 ActiveRecord (strong_migrations port)

Tree-sitter-ruby query catches every dangerous AR migration call:
```
(method_call method: (identifier) @m
  (#match? @m "^(add_column|remove_column|add_index|change_column|change_column_null|add_foreign_key|add_check_constraint|add_reference)$"))
```
Apply strong_migrations' "dangerous"/"safe" classification.

### A.6 Knex / Sequelize / golang-migrate / dbmate / Diesel / Refinery

| Rule | Bad pattern |
|------|-------------|
| `knex-missing-down` | `exports.down = function() {}` empty |
| `sequelize-queryInterface-no-reverse` | `down: async (queryInterface) => {}` empty |
| `golang-migrate-missing-down` | `..._up.sql` without sibling `..._down.sql` |
| `dbmate-pair-mismatch` | same |
| `diesel-empty-down-sql` | `migrations/*/down.sql` empty |
| `refinery-out-of-order-version` | file numbers regress between PRs |

### A.7 Sqitch / Atlas (declarative)

| Rule | Bad pattern |
|------|-------------|
| `sqitch-change-no-revert` | `sqitch.plan` lists change without `revert/<name>.sql` |
| `sqitch-no-verify-script` | missing `verify/<name>.sql` |
| `atlas-destructive-diff-unguarded` | `atlas migrate apply` in CI without `--lint-review WARNING` |
| `atlas-policy-missing` | `atlas.hcl` without `lint { destructive {…} }` block |

### A.8 MySQL hot-table ALTER → recommend gh-ost / pt-osc

For MySQL stacks, any `ALTER TABLE` on a known "hot tables" allow-list should
trigger `migration-needs-osc` (Severity High, Effort Small) pointing at
gh-ost / pt-online-schema-change.

## PART B — Connection Pool Configuration Anti-Patterns

### B.0 Universal sizing rule

From HikariCP wiki: `connections = ((core_count * 2) + effective_spindle_count)`.
Pool sizes above ~20–30 per app instance are almost always wrong.

**The N-connection math (cross-file static check):**
`effective_max = max_pool_size × app_workers × replica_count`
If this exceeds DB `max_connections` (both numbers from the same repo via
Docker compose / k8s / Terraform), emit `pool-exhaustion-risk` (High).

### B.1 HikariCP (Java)

| Rule | Bad pattern |
|------|-------------|
| `hikari-pool-too-large` | `maximumPoolSize > 20` per node |
| `hikari-min-idle-mismatch` | `minimumIdle != maximumPoolSize` (Hikari author recommends fixed-size) |
| `hikari-no-leak-detection` | `leakDetectionThreshold = 0` in prod profile |
| `hikari-maxlifetime-gt-wait-timeout` | `maxLifetime > MySQL wait_timeout` (default 28800s) |
| `hikari-connection-timeout-too-short` | `connectionTimeout < 250` ms |

### B.2 SQLAlchemy (Python)

| Rule | Bad pattern |
|------|-------------|
| `sqla-pool-pre-ping-false` | `create_engine(..., pool_pre_ping=False)` or omitted |
| `sqla-pool-recycle-missing-mysql` | MySQL DSN with no `pool_recycle` |
| `sqla-nullpool-with-web-app` | `poolclass=NullPool` in non-serverless context |
| `sqla-singletonthreadpool-in-async` | `SingletonThreadPool` with `asyncio` in same module |
| `sqla-overflow-exceeds-db-limit` | `pool_size + max_overflow > DB max_connections` |

### B.3 Django

| Rule | Bad pattern |
|------|-------------|
| `django-conn-max-age-zero` | `'CONN_MAX_AGE': 0` (default) in WSGI prod settings |
| `django-conn-health-checks-off` | `CONN_HEALTH_CHECKS = False` with `CONN_MAX_AGE > 0` |
| `django-atomic-requests-true` | `'ATOMIC_REQUESTS': True` globally on high traffic |

### B.4 psycopg / asyncpg

| Rule | Bad pattern |
|------|-------------|
| `asyncpg-pool-defaults-only` | bare `asyncpg.create_pool(dsn)` — uses 10/10/300s defaults |
| `psycopg3-no-pool` | `psycopg.connect(...)` in request handler instead of `psycopg_pool.ConnectionPool` |

### B.5 node-postgres / mysql2 / Mongoose

| Rule | Bad pattern |
|------|-------------|
| `pg-pool-no-explicit-max` | `new Pool({})` — default 10 |
| `pg-idle-timeout-zero` | `idleTimeoutMillis: 0` with serverless |
| `mysql2-default-connectionLimit` | `mysql2.createPool({})` — default 10 |
| `mongoose-maxpoolsize-default-100` | `mongoose.connect(uri)` with no `maxPoolSize` — default 100 |
| `mongoose-waitQueueTimeoutMS-missing` | no `waitQueueTimeoutMS` (default infinite wait) |

### B.6 Prisma

| Rule | Bad pattern |
|------|-------------|
| `prisma-no-connection-limit-serverless` | DSN lacks `?connection_limit=1` when on Vercel/Lambda |
| `prisma-pool-timeout-too-low` | `?pool_timeout=0` (disables timeout) |
| `prisma-default-connection-limit` | DSN omits — Prisma uses `num_cpus * 2 + 1` |

### B.7 TypeORM / Sequelize / GORM / sqlx / pgx / .NET

| Rule | Bad pattern |
|------|-------------|
| `typeorm-pool-defaults` | `extra: {}` empty in DataSource |
| `sequelize-pool-max-default-5` | `pool: { max: 5 }` is default — flag in prod |
| `gorm-no-setmaxopenconns` | no `db.SetMaxOpenConns(...)` after `gorm.Open` |
| `gorm-no-setconnmaxlifetime` | no `SetConnMaxLifetime` |
| `sqlx-rust-default-pool` | `PgPool::connect(url)` (default max=10) instead of `PgPoolOptions::new().max_connections(N)` |
| `pgx-default-maxconns` | `pgxpool.New(...)` without `Config.MaxConns` |
| `dotnet-max-pool-size-too-high` | conn-string `Max Pool Size=500` |

### B.8 PgBouncer / Supabase pooler

| Rule | Bad pattern |
|------|-------------|
| `pgbouncer-transaction-mode-with-prepared-statements` | `pool_mode=transaction` AND app uses prepared statements without `max_prepared_statements>0` or `prepareThreshold=0` |
| `pgbouncer-statement-mode-with-transactions` | `pool_mode=statement` with code using `BEGIN`/`COMMIT` |
| `supabase-pooler-port-6543-with-prepare` | DSN uses port 6543 without `?pgbouncer=true&statement_cache_size=0` |

## PART C — Transaction Management Anti-Patterns

### C.1 Long-running transactions (the #1 production killer)

| Rule | Bad pattern |
|------|-------------|
| `tx-http-call-in-transaction` | HTTP/gRPC/Redis/SQS/sleep call lexically inside `@transaction.atomic` / `with session.begin():` / `@Transactional` / `db.Transaction(ctx, ...)` / `EntityManager.transaction()` |
| `tx-external-api-anti-outbox` | Same, specifically for non-idempotent POST/PUT — recommend outbox pattern |

### C.2 Transaction per loop iteration

| Rule | Bad pattern |
|------|-------------|
| `tx-per-loop-iteration` | `for x in xs: with transaction.atomic(): ...` (1 round-trip/item) |
| `tx-per-loop-iteration-java` | `for(...) { tx.execute(...); }` inside `TransactionTemplate` |

Fix: hoist transaction outside, or batch via `bulk_*`.

### C.3 Spring @Transactional cluster

| Rule | Bad pattern |
|------|-------------|
| `spring-transactional-on-private` | `@Transactional` on `private` method (silently disabled in CGLIB proxy pre-Spring 6) |
| `spring-transactional-self-invocation` | method A calls `this.b()` where `b` is `@Transactional` — proxy bypassed |
| `spring-transactional-checked-exception-swallow` | `@Transactional` (no `rollbackFor`) declares `throws SomeCheckedException` |
| `spring-transactional-requires-new-self-call` | self-call with `Propagation.REQUIRES_NEW` |
| `spring-transactional-read-only-missing` | `@Transactional` (writable) on method that only does `findBy*` — should be `readOnly = true` |
| `spring-transactional-on-async` | `@Transactional` + `@Async` on same method — different thread, tx not propagated |

### C.4 Django / Rails ATOMIC_REQUESTS

| Rule | Bad pattern |
|------|-------------|
| `django-atomic-requests-true-high-traffic` | `'ATOMIC_REQUESTS': True` in settings |
| `rails-transaction-requires-new` | `Model.transaction(requires_new: true)` outside outer transaction (no-op) |

### C.5 Read-modify-write race

| Rule | Bad pattern |
|------|-------------|
| `rmw-race-django` | `obj = M.objects.get(id=x); obj.counter += 1; obj.save()` outside `select_for_update()` + `atomic()` |
| `rmw-race-sqlalchemy` | `obj = session.get(M, id); obj.x += 1; session.commit()` outside `with_for_update()` |
| `rmw-race-typeorm` | `const e = await repo.findOne(id); e.x++; await repo.save(e);` without `setLock('pessimistic_write')` |

Fix: `select_for_update`, `F('counter') + 1`, or `@Version`.

### C.6 Missing optimistic locking

| Rule | Bad pattern |
|------|-------------|
| `jpa-no-version-on-mutable-entity` | `@Entity` with mutable fields and no `@Version` |
| `eloquent-no-version-attribute` | Laravel model with no `version` column |
| `sqlalchemy-no-version-id-col` | SQLAlchemy `Mapper.version_id_col` not set on hot mutable models |

### C.7 SELECT FOR UPDATE recognition (positive)

| Rule | Recognize |
|------|---|
| `tx-select-for-update-no-atomic` | `Model.objects.select_for_update().get(...)` not inside `with transaction.atomic():` — Django raises `TransactionManagementError` |
| `tx-skip-locked-for-queue-ok` | `select_for_update(skip_locked=True)` recognized as queue pattern — suppress other fires |

### C.8 Transaction isolation levels

| Rule | Bad pattern |
|------|-------------|
| `tx-read-uncommitted-application` | `SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED` |
| `tx-serializable-on-hot-path` | `Isolation.SERIALIZABLE` on a method annotated `@GetMapping` |
| `tx-snapshot-vs-rr-confusion` | `IsolationLevel.REPEATABLE_READ` in code switching between PG and MySQL drivers |

### C.9 Deadlock-prone

| Rule | Bad pattern |
|------|-------------|
| `tx-multi-lock-in-loop` | ≥2 `select_for_update`/`lock()` inside same loop body |
| `tx-lock-order-inversion` | two transactional methods lock same two tables in opposite orders |

### C.10 Autocommit / connection leaks

| Rule | Bad pattern |
|------|-------------|
| `tx-autocommit-false-no-commit` | `conn.autocommit = False` without matching `commit()` on all paths |
| `tx-begin-without-context-manager` | `session.begin()` not used as context manager + no `try/finally` |

### C.11 Outbox pattern violations

| Rule | Bad pattern |
|------|-------------|
| `tx-kafka-publish-in-transaction` | `kafkaTemplate.send(...)` inside transactional scope without outbox table |
| `tx-stripe-charge-in-transaction` | `stripe.PaymentIntent.create` inside transaction — non-rollbackable |
| `tx-email-send-in-transaction` | `send_mail` inside transaction |

### C.12 Existing analyzers to consult

- `flake8-django` (MIT) — no transaction rules; clear gap.
- `eslint-plugin-typeorm` (MIT) — query-builder only.
- PMD/Checkstyle — no `@Transactional` rules.
- No mature open analyzer covers C.1, C.2, or C.11 — **drift can lead here.**

## Cross-cutting implementation notes

1. **Migration file discovery**: single resolver per ecosystem — `alembic/versions/*.py`, `db/migrate/*.rb`, `*/migrations/*.py`, `prisma/migrations/**/*.sql`, `*V[0-9]+__*.sql`, `db/changelog/**`, `migrations/*_up.sql`.
2. **DSN parsing utility**: extract `?pool_size=`, `?connection_limit=`, port number once; reuse.
3. **The "math" rule**: pool_size × workers vs DB max_connections — actionable when both live in repo.
4. **Severity defaults**: squawk's rule list maps cleanly to High/Medium/Low.
5. **License**: GPL-3.0 squawk = re-implement, don't copy. MIT (strong_migrations, flake8-django, pgroll, gh-ost, atlas) = port verbatim with attribution.

## Recommended ordering (Trivial → Large)

1. **Trivial**: `typeorm-synchronize-true`, `gorm-automigrate-prod`, `hikari-pool-too-large`, `mongoose-maxpoolsize-default-100`, `django-conn-max-age-zero` — single-call/single-config.
2. **Small**: full squawk port for raw SQL migration files; strong_migrations port for Ruby; Spring `@Transactional` on private + self-invocation.
3. **Medium**: `tx-http-call-in-transaction`, `tx-per-loop-iteration`, `tx-kafka-publish-in-transaction` — scope walker plus call classification.
4. **Large**: cross-file pool-math; Alembic multi-head; lock-order-inversion.
